좋은 GitHub 포트폴리오 만드는 법

GitHub 포트폴리오는 저장소 수보다 구조가 더 중요합니다. 처음 보는 사람이 핵심 프로젝트와 작업 흐름을 빠르게 이해할 수 있어야 설득력이 생깁니다.

핵심만 보기
  • 대표 프로젝트 몇 개를 먼저 정리하는 편이 더 읽기 좋습니다.
  • README는 결과보다 맥락과 선택 이유를 설명해야 합니다.
  • 저장소 흐름과 최근 활동도 포트폴리오 인상에 영향을 줍니다.

대표 프로젝트를 먼저 고르세요

모든 저장소를 같은 비중으로 보여 주면 핵심이 흐려질 수 있습니다. 보여 주고 싶은 프로젝트 몇 개를 먼저 정리하고, 나머지는 보조 자료처럼 두는 편이 더 깔끔합니다.

특히 포트폴리오에서는 "많이 만들었다"보다 "무엇을 잘 보여주고 있는가"가 더 중요합니다. 대표 프로젝트마다 역할이 분명하면 읽는 사람도 빠르게 이해할 수 있습니다. 예를 들어 한 저장소는 제품 감각, 다른 저장소는 백엔드 구조, 또 다른 저장소는 협업 흐름을 보여주는 식으로 구성이 나뉘면 인상이 더 또렷해집니다.

README는 맥락을 설명해야 합니다

무엇을 만들었는지만 적는 것보다 왜 만들었는지, 어떤 문제를 해결하려 했는지까지 함께 적는 문서가 훨씬 설득력 있습니다. 개인 프로젝트일수록 이 차이가 크게 보입니다.

README는 결과물을 자랑하는 글보다, 처음 보는 사람이 "이 프로젝트를 왜 만들었고 어떤 선택을 했는지" 이해할 수 있게 돕는 안내서에 가깝습니다. 구현 배경과 기술 선택 이유가 보이면 같은 기능의 프로젝트라도 훨씬 성숙하게 읽힙니다.

활동 흐름도 함께 보입니다

GitHub는 결과물뿐 아니라 작업 습관도 드러나는 공간입니다. 오랫동안 방치된 대표 저장소는 README를 다듬거나 간단한 개선 기록을 남겨 흐름을 다시 살려 두는 것도 도움이 됩니다.

포트폴리오 점검 포인트
  • 대표 저장소의 최근 업데이트 시점이 너무 오래되지 않았는지 확인합니다.
  • 커밋 메시지, PR 설명, 이슈 메모 중 최소 하나는 작업 맥락이 보이게 남겨 둡니다.
  • 대표 프로젝트와 최근 활동 흐름이 완전히 분리돼 보이지 않도록 관리합니다.

설명 가능한 저장소가 더 강합니다

프로젝트 목적, 주요 기능, 기술 선택 이유, 실행 방법을 스스로 설명할 수 있어야 저장소가 더 설득력 있게 읽힙니다. 정돈된 몇 개의 저장소가 산만한 많은 저장소보다 더 좋은 인상을 줍니다.

특히 채용이나 협업 검토 상황에서는 "무엇을 만들었다"보다 "무엇을 설명할 수 있는가"가 더 중요할 때가 많습니다. 화면, API, 데이터 구조, 배포 방식, 트러블슈팅 중 어느 부분이든 한두 문장으로 또렷하게 설명할 수 있는 저장소가 훨씬 강합니다.

자주 생기는 오해도 줄여 보세요

저장소 수가 적다고 약한 포트폴리오는 아닙니다. 반대로 저장소가 많아도 설명이 부족하거나 역할이 겹치면 오히려 핵심이 흐려질 수 있습니다. 포트폴리오는 수집보다 편집에 가깝다는 점을 기억하면 정리 방향을 잡기 쉽습니다.
